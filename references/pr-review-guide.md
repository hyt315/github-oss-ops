## 审查流程

```
读取 PR 元信息（标题、描述、关联 Issue）
    ↓
获取改动文件列表（pull_request_read, method: get_files）
    ↓
检查 CI/CD 状态和可合并状态
    ↓
分析改动类型和范围
    ↓
逐项检查（见下方清单）
    ↓
生成审查摘要 → 展示给维护者 → 确认后提交
```

## 自动检查清单

以下项目可以通过分析 PR 数据自动判断：

### 基础信息

| 检查项 | 方法 |
|--------|------|
| 是否关联 Issue | 检查 PR 描述中是否有 `#number`、`closes #`、`fixes #` |
| 改动范围 | 统计改动文件数和行数（+X/-Y） |
| 改动类型 | 根据文件路径判断：代码文件 / 配置文件 / 文档 / 测试 |
| 是否包含测试 | 检查是否有 `test/` 或 `*.test.*` 文件的改动 |
| 是否更新了文档 | 检查是否有 `README`、`docs/`、`*.md` 的改动 |
| 贡献者身份 | 首次贡献者 / 活跃贡献者（通过 commit 历史判断） |
| CI/CD 检查状态 | 通过 `pull_request_read`（method: `get_status`）获取检查状态（pending/success/failure） |
| 可合并状态 | 检查 `mergeable` 和 `mergeable_state` 字段（clean/blocked/dirty/unknown） |
| 分支冲突 | 检查是否与目标分支有冲突（`mergeable_state` 为 `dirty` 时有冲突） |
| CODEOWNERS 审查 | 检查是否已获得 CODEOWNERS 中定义的必要审查 |

### 提交质量

| 检查项 | 说明 |
|--------|------|
| 提交数量 | 过多的小提交可能需要 squash |
| 提交信息 | 是否遵循 Conventional Commits（如适用） |
| 分支来源 | 是否从正确的分支 fork/branch |

## 代码质量检查

针对代码类 PR 的具体检查：

### 通用检查

| 问题 | 检测方法 |
|------|----------|
| 硬编码值 | 搜索 API Key、URL、路径等硬编码 |
| 敏感信息 | 检查是否提交了 token、password、.env 等 |
| 大文件 | 检查是否提交了二进制文件或大文件 |
| 未使用的导入 | 检查新增的 import 是否被使用 |
| 错误处理 | 新增的 try/catch 或错误处理是否完整 |
| 类型安全 | TypeScript/类型注解是否完整 |

### 语言特定检查

| 语言 | 检查项 |
|------|--------|
| JavaScript/TypeScript | `var` 使用（应用 `let/const`）、`==` vs `===` |
| Python | 未使用的导入、`print()` 调试语句、缺少类型注解 |
| Rust | `unwrap()` 使用（应处理错误）、`clone()` 过度使用 |
| Go | 未处理的 error、context 传递 |

## 安全审查

| 风险 | 检测方法 |
|------|----------|
| 敏感信息泄露 | 搜索 token、key、password、secret、credential |
| 依赖漏洞 | 检查 `package.json` / `requirements.txt` / `Cargo.toml` 的新依赖 |
| 注入风险 | 搜索 SQL 拼接、`eval()`、`exec()` |
| 路径遍历 | 检查文件操作中是否有用户输入直接拼接路径 |
| 权限提升 | 检查是否新增了 sudo、admin 相关代码 |

## 常见 PR 问题和处理

| 问题 | 处理建议 |
|------|----------|
| PR 过大（>500 行改动） | 建议拆分为多个小 PR，更容易审查 |
| PR 包含不相关改动 | 建议移除无关改动，保持 PR 聚焦 |
| 没有测试 | 如果改动涉及逻辑，要求补充测试 |
| 破坏性变更 | 确认是否更新了版本号、文档、迁移指南 |
| 与主线冲突 | 提醒提交者 rebase 或 merge 最新主线 |
| 代码风格不一致 | 指出具体不一致的位置，或引用 .editorconfig |
| 缺少描述 | 提醒提交者补充 PR 描述（做了什么、为什么、怎么测试） |
| CI/CD 检查失败 | 先查看失败原因，区分是代码问题还是环境问题，不要在 CI 红灯时合并 |
| Dependabot PR | 检查依赖变更范围，确认无破坏性变更后合并；安全更新优先处理 |
| 首次贡献者 PR | 额外耐心，提供具体指导，避免直接拒绝 |

### Dependabot PR 审查要点

Dependabot 自动创建的 PR 需要特别关注：

| 检查项 | 说明 |
|--------|------|
| 变更类型 | 版本更新（major/minor/patch）还是安全修复 |
| 破坏性风险 | major 版本更新可能有 API 破坏性变更 |
| CI 状态 | 确认 CI 检查通过 |
| 变更范围 | 检查 `package-lock.json` / `yarn.lock` 等锁文件变更是否合理 |
| CHANGELOG | 查看依赖的 CHANGELOG 确认无破坏性变更 |

> 安全更新（Security Update）应优先审查和合并，不要长期搁置。

## 审查意见模板

### 通过（Approve）

```markdown
## 审查结论：通过

### 总结
{一句话概括 PR 做了什么}

### 检查清单
- [x] 改动范围合理（{N} 文件，+{X}/-{Y}）
- [x] 代码风格一致
- [x] 包含测试
- [x] 无安全问题
- [x] 文档已更新

### 评价
{具体的正面评价}
```

### 需要修改（Request Changes）

```markdown
## 审查结论：需要修改

### 总结
{一句话概括 PR 做了什么}

### 需要修改
1. **{文件名}:{行号}** — {具体问题和建议}
2. **{文件名}:{行号}** — {具体问题和建议}

### 建议（非必须）
- {可选的改进建议}

### 检查清单
- [x] 改动范围合理
- [ ] 代码风格一致（具体问题）
- [ ] 包含测试（缺少 {某功能} 的测试）
- [x] 无安全问题
```

### 拒绝（Request Changes / Close）

```markdown
## 审查结论：暂不合并

### 原因
{具体原因，客观不带主观评价}

### 建议
{给提交者的下一步建议，如"建议先在 Issue 中讨论设计方案"}
```

## 合并决策树

```
PR 提交
    ↓
CI/CD 检查是否通过？
    ├─ 否 → Request Changes，指出 CI 失败原因
    └─ 是 ↓
是否有冲突？
    ├─ 是 → 提醒提交者 rebase
    └─ 否 ↓
CODEOWNERS 审查是否完成？
    ├─ 否 → 等待必要审查
    └─ 是 ↓
代码审查结果？
    ├─ 有阻断问题 → Request Changes
    ├─ 有建议但非阻断 → Approve + 评论建议
    └─ 无问题 ↓
合并方式？
    ├─ 单 commit 改动 → Squash merge
    ├─ 多 commit 有意义 → Merge commit
    └─ 需要保留提交历史 → Rebase merge
```

> **合并方式建议**：开源项目推荐 Squash merge（保持主线整洁），在 PR 描述中保留完整信息。
